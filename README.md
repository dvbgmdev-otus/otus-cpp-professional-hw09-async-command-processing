# ДЗ-09. Многопоточная асинхронная обработка команд

Проект реализует асинхронную обработку пакетных команд из задания OTUS C++ Professional.

Основной результат работы - библиотека `libasync.so` с публичным интерфейсом `async.h`.
Дополнительно собирается демонстрационная консольная утилита `bulk`, которая читает данные из
`stdin`, передает их в библиотеку и показывает работу асинхронной обработки.

## Что делает программа

Входной поток состоит из команд, разделенных переводом строки.
Команды группируются в блоки и выводятся:

- в консоль;
- в файлы `bulk<timestamp>_<index>.log`.

Пример вывода:

```text
bulk: cmd1, cmd2, cmd3
bulk: cmd4, cmd5
```

Поддерживаются два режима группировки:

- статический блок - если фигурных скобок нет, команды группируются по размеру блока,
  переданному при запуске;
- динамический блок - команды между `{` и `}` образуют отдельный блок, который не зависит
  от размера статического блока.

## Асинхронная модель

При работе используются четыре логических потока:

- `main` - поток вызывающего кода, который передает входные данные;
- `log` - поток вывода готовых блоков в консоль;
- `file1` - первый поток записи блоков в файлы;
- `file2` - второй поток записи блоков в файлы.

Готовый блок команд публикуется в две очереди:

- очередь консольного вывода обрабатывается одним потоком `log`;
- очередь файлового вывода обрабатывается двумя потоками `file1` и `file2`.

Имя файла сохраняет timestamp блока и получает дополнительный числовой постфикс.
Это гарантирует уникальность имен при параллельной записи.

## Публичный API библиотеки

Заголовочный файл библиотеки:

```cpp
#include <async.h>
```

Интерфейс:

```cpp
namespace async {

using handle_t = void*;

handle_t connect(std::size_t bulk);
void receive(handle_t handle, const char* data, std::size_t size);
void disconnect(handle_t handle);

}  // namespace async
```

Сценарий использования:

```cpp
auto handle = async::connect(3);

async::receive(handle, buffer, size);

async::disconnect(handle);
```

`handle_t` - непрозрачный указатель на внутренний контекст обработки.
Внешний код не должен интерпретировать это значение, он только передает его обратно в
`receive()` и `disconnect()`.

Можно создать несколько независимых контекстов через несколько вызовов `connect()`.
Вызовы `receive()` для разных контекстов не смешивают незавершенные блоки команд.

## Структура проекта

```text
src/
  async/     публичный API async и асинхронный runtime
  core/      синхронная доменная логика: парсер, форматирование, вывод, CLI
  main/      демонстрационная утилита bulk
tests/
  unit/      unit-тесты
  coverage/  сборка отчета покрытия
cmake/       общие CMake-модули проекта
scripts/     инфраструктурные shell-скрипты
```

Ключевые компоненты:

- `src/async/async.h` - публичный заголовок библиотеки;
- `src/async/async.cpp` - реализация функций `connect()`, `receive()`, `disconnect()`;
- `src/async/async_context.*` - контекст одного подключения;
- `src/async/async_runtime.*` - общий runtime с потоками вывода;
- `src/async/blocking_queue.h` - блокирующая очередь для обмена между потоками;
- `src/core/bulk_processor.*` - формирование блоков команд;
- `src/core/io/bulk_writer.*` - вывод блоков в консоль и файлы;
- `src/main/main.cpp` - пример клиентского кода, использующего библиотеку.

## Требования

Для локальной сборки через проектные скрипты нужен Docker.
Скрипты запускают сборку внутри образа `otus_builder`.

Для нативной сборки нужны:

- CMake 3.12 или новее;
- компилятор C++ с поддержкой C++14;
- доступ к GitHub для загрузки GoogleTest через CMake FetchContent при конфигурации тестов;
- `lcov` для отчета покрытия;
- `cppcheck` и `clang-tidy` для статического анализа.

## Сборка

Основная команда сборки:

```bash
./prj build
```

Скрипт сам определяет, выполняется он на хосте или внутри Docker.
На хосте сборка будет запущена внутри контейнера.

Нативный эквивалент:

```bash
cmake -S . -B build
cmake --build build
```

## Запуск

Запуск через проектный скрипт:

```bash
printf "cmd1\ncmd2\ncmd3\ncmd4\n" | ./prj run 3
```

Запуск собранного бинарного файла напрямую:

```bash
printf "cmd1\ncmd2\ncmd3\ncmd4\n" | ./build/bin/bulk 3
```

Ожидаемый вывод:

```text
bulk: cmd1, cmd2, cmd3
bulk: cmd4
```

Пример динамического блока:

```bash
printf "cmd1\n{\ncmd2\ncmd3\n}\ncmd4\n" | ./build/bin/bulk 3
```

Ожидаемый вывод:

```text
bulk: cmd2, cmd3
bulk: cmd1, cmd4
```

## Тесты

Запуск всех тестов:

```bash
./prj test
```

Тесты проверяют:

- парсинг аргументов командной строки;
- формирование статических и динамических блоков;
- форматирование вывода;
- запись в консоль и файлы;
- работу блокирующей очереди;
- контекст async;
- публичный API библиотеки `async`.

## Покрытие

Запуск отчета покрытия:

```bash
./prj cov
```

После выполнения отчет формируется в:

```text
build/out/index.html
```

## Статический анализ

Cppcheck:

```bash
./prj cppcheck
```

clang-tidy:

```bash
./prj tidy
```

У `clang-tidy` есть уровни проверки:

- `--light` - минимальный малошумный анализ, используется по умолчанию;
- `--medium` - расширенный анализ с дополнительными `bugprone` и `performance` проверками;
- `--strict` - жесткий аудит, может давать много предупреждений.

Примеры:

```bash
./prj tidy --light
./prj tidy --medium
./prj tidy --strict
./prj tidy --medium -- src/async/async.cpp
```

## Пакет

Задание требует пакет `async`, в который входят:

- `libasync.so` - библиотека;
- `async.h` - публичный заголовочный файл;
- `bulk` - демонстрационный исполняемый файл.

Сборка пакета после конфигурации проекта:

```bash
docker run --rm -v "$PWD":/app -w /app otus_builder \
  cmake --build build --target package
```

Проверка содержимого deb-пакета выполняется после сборки пакета:

```bash
docker run --rm -v "$PWD":/app -w /app otus_builder \
  sh -c 'dpkg-deb --contents build/*.deb'
```

Можно выполнить оба шага одной командой:

```bash
docker run --rm -v "$PWD":/app -w /app otus_builder \
  sh -c 'cmake --build build --target package && dpkg-deb --contents build/*.deb'
```

Ожидаемые установочные пути:

```text
/usr/bin/bulk
/usr/include/async.h
/usr/lib/libasync.so
```
